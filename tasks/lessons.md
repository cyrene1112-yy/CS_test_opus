# Lessons Learned

## L001 · 大文件一次性重写导致 Error writing file / 响应超时

### 现象

对 ~1500 行的 `index.html` 进行多人改造时，试图用 `Write` 工具一次性输出完整重写版本，导致响应体过长，写入失败并中断会话。

### 根因

`Write` 工具（及 LLM 单次响应）对输出长度有限制。当改动覆盖全文件且文件较大时，一次性生成完整内容会超出限制。

### 规则

**禁止对超过 200 行的文件使用 `Write` 全量重写。**

改用 `Edit` 工具做外科手术式的局部替换：
1. 先 `Read` 文件，确认要改的代码段
2. 每次 `Edit` 只改一个逻辑单元（一个函数 / 一个代码块）
3. 多个不相关的改动可以并行发出多个 `Edit`，但每个 `Edit` 保持单一职责

### 适用场景判断

| 情况 | 做法 |
|------|------|
| 新建文件（< 200 行） | `Write` 可以 |
| 已有文件，局部改动 | `Edit` |
| 已有文件，改动 > 50% | 拆成多个步骤，逐步 `Edit`，每步用户确认 |

### 参考

本次改造按 9 个步骤分批执行，每步完成后用户确认，顺利完成。与一次性重写相比，分步执行还带来额外好处：每步可独立验证，出错范围小、回滚成本低。
